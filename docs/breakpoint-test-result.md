### Mock 서버 특성

- 요청마다 **지수 분포 랜덤 지연** (평균 4s, 최대 30s)
- **1/3 확률로 500 에러** 반환 (의도적 설계)
- Tomcat `max-threads` 기본값: **200**
- Tomcat `accept-count`: 100

---

## 테스트 설계

VU(Virtual User, 동시 요청 수)를 단계적으로 올려가며 서버 한계를 탐색하는 Breakpoint Test.

```
50 VU → 100 VU → 200 VU → 300 VU → 400 VU
(각 단계 30s ramp-up + 1분 hold)
```

> **VU란?** k6에서 동시에 HTTP 요청을 계속 보내는 가상 클라이언트 수.
> VU=200이면 200명이 쉬지 않고 계속 요청을 보내는 상태.

## 핵심 발견

### 1. VU가 400까지 올라가도 avg 응답시간이 4s로 유지됨

```
VU  50 → avg 3.8s
VU 100 → avg 3.7s
VU 200 → avg 3.9s  ← 스레드 200으로 포화됐는데도
VU 300 → avg 4.1s  ← 응답시간 거의 변화 없음
VU 400 → avg 4.0s  ← 동일
```

**이유**: Tomcat NIO는 스레드 풀이 꽉 찬 상태에서도 대기 중인 요청을 스레드 하나가 끝나는 즉시 처리한다.
200개 스레드 중 가장 먼저 끝나는 것을 기다리는 평균 시간은 극히 짧아서 (4s ÷ 200 ≈ 0.02s)
avg 응답시간에 거의 영향을 주지 않는다.

**실질 한계는 응답시간이 아니라 처리량(TPS)이었다.**

### 2. TPS가 200 VU에서 상한선에 도달

| VU | TPS (10s 평균) | 상태 |
|----|--------------|------|
| 50 | 14/s | 여유 |
| 100 | 27/s | 여유 |
| **200** | **50/s** | **포화 도달** |
| 300 | 50/s | 변화 없음 |
| 400 | 51/s | 변화 없음 |

VU 200에서 TPS 50/s에 도달한 후 300, 400으로 올려도 증가 없음.

이는 이론값과 정확히 일치한다:

```
Tomcat max-threads(200) ÷ 평균 응답시간(4s) = 50 TPS
```

### 3. 응답시간 폭등은 과부하가 아니라 cooldown 때 발생

테스트 중에는 avg가 4s로 안정적이다가, **VU가 0으로 내려가는 cooldown 구간에서만 avg가 최대 28s까지 폭등**한다.

**이유**: 지수 분포 특성상 15~30초짜리 롱테일 요청이 소수 존재한다.
테스트 중에는 짧은 요청들이 계속 완료되면서 평균을 4s로 유지시켜주지만,
VU가 줄어들고 신규 요청이 없어지면 남아있는 롱테일 요청만 처리되어 평균이 튄다.

### 4. 활성 스레드 감소 + 응답시간 증가가 동시에 발생하는 이유

직관적으로 이상해 보이지만 다음과 같은 이유다:

```
cooldown → k6가 새 요청 보내는 것을 줄임
         → 스레드들이 처리 완료 후 새 일 없이 idle
         → busy_threads 감소

동시에 →  아직 처리 중인 롱테일 요청(20~30초짜리)이 남아있음
         → 이 요청들이 완료되면서 avg 응답시간 끌어올림
```

**스레드 내려가는 이유 = 새 요청이 안 들어와서**
**응답시간 올라가는 이유 = 지수 분포 롱테일 요청이 마무리되면서**

## 에러율 32.88%에 대하여

```
http_req_failed: 32.88%  ≈  서버 설계의 1/3 랜덤 에러율
```

이 에러는 **서버 과부하로 인한 거절이 아니다.**
코드를 보면 500 에러도 동일한 지수 분포 지연(Thread.sleep)을 거친 후 발생한다.
즉, 32.88%는 서버가 의도적으로 설계한 랜덤 실패율이며, 커넥션 거절이나 타임아웃 에러는 0건이다.

## 결론

### Mock 서버의 동시 처리 한계 = 200 VU = Tomcat max-threads(200)

| 지표 | 한계 지점 | 근거 |
|------|----------|------|
| TPS | 200 VU (50 req/s) | 이후 VU 증가해도 TPS 증가 없음 |
| 활성 스레드 | 200 VU (200개 포화) | Prometheus 실측값 |
| 응답시간 | **의미있는 변화 없음** | VU 50~400 전 구간 avg 4s 유지 |

### 실제 서비스 설계에 주는 시사점

이 결과는 **알림 발송 서비스** 설계에서 중요한 시사점을 준다.

실제 외부 알림 API(이메일, Teams 등)도 응답이 느리고 실패가 잦은 상황을 가정하면,
동기 방식으로 Tomcat 스레드가 외부 API 응답을 기다리는 구조는 **스레드 풀 소진**으로 이어진다.

응답시간 자체는 4s로 유지되더라도 **처리량이 50 TPS에서 막히는** 구조적 한계가 있으며,
이를 해결하려면:

- **비동기 처리** 도입 (응답 반환 후 별도 스레드에서 발송)
- **메시지 큐** 활용 (발송 요청을 큐에 넣고 별도 워커가 처리)
- Tomcat `max-threads` 튜닝 (임시방편, 근본 해결책 아님)

중 하나를 선택해야 한다.

## 참고: k6 threshold 설정 문제

현재 `breakpoint-test.js`의 threshold:
```javascript
http_req_duration: ['p(95)<3000', 'p(99)<5000'],  // 3s, 5s
http_req_failed: ['rate<0.05'],                     // 5%
```

이 서버에서는 처음부터 통과 불가능한 기준이다.
- 평균 4s 지수 분포의 자연스러운 p(95) ≈ 12s → `p(95)<3s` 불가
- 설계된 에러율 33% → `rate<0.05` 불가

Breakpoint Test 목적에 맞는 기준으로 수정한다면:
```javascript
http_req_duration: ['p(95)<15000'],  // 자연 분포 p(95)=12s보다 여유있게
http_req_failed: ['rate<0.40'],      // 설계 에러율 33% 위로
```
