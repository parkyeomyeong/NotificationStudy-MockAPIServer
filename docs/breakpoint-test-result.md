# Mock 서버 부하 테스트 결과 분석

> **테스트 목적**: 알림 발송 미들웨어 설계 시 Tomcat 쓰레드 풀 크기를 얼마나 할당해야 하는지 판단하기 위한 임계치 탐색 (Breakpoint Test)

---

## 1. 배경 및 가설

알림 발송 서비스는 외부 API(이메일, Teams 등)를 호출하는 구조 특성상 **응답 대기 시간이 길다**.
Tomcat 기반 동기 처리 구조에서는 이 대기 시간 동안 쓰레드가 점유되므로, 동시 요청이 늘어나면 쓰레드 풀이 빠르게 고갈된다.

이를 실제로 확인하기 위해 외부 API 응답을 모사하는 Mock 서버를 구성하고 부하를 단계적으로 올려가며 처리 한계를 측정했다.

**사전 이론값**:
```
최대 TPS = Tomcat max-threads ÷ 평균 응답시간
         = 200 ÷ 4s
         = 50 TPS
```

이론상 200 VU에서 TPS 상한에 도달할 것으로 예측했고, 이를 실측으로 검증하는 것이 이번 테스트의 핵심이었다.

---

## 2. 테스트 환경

| 항목 | 내용 |
|------|------|
| 부하 도구 | k6 |
| 대상 서버 | Spring Boot Mock API Server (192.168.0.2:8081) |
| 모니터링 | Prometheus + Grafana |
| 테스트 스크립트 | `k6/breakpoint-test.js` |

### Mock 서버 설계 의도

실제 외부 알림 API의 동작을 모사하기 위해 다음 두 가지를 의도적으로 설계했다.

**지연 시뮬레이션**: 지수 분포(λ=0.25, 평균 4s, 최대 30s)

```java
double exponentialSeconds = -Math.log(1 - uniform) / LAMBDA;
```

실제 외부 API는 대부분 빠르게 응답하지만 간헐적으로 수십 초 걸리는 경우가 있다. 지수 분포는 이 특성을 잘 모사한다. 부하 없을 때 이론적 응답시간 분포는 다음과 같다:

| 구간 | 이론값 |
|------|--------|
| Median (P50) | ≈ 2.8s |
| P95 | ≈ 12s |
| P99 | ≈ 18s |

**에러 시뮬레이션**: `SUCCESS / FAILURE / ERROR` 중 균등 랜덤 선택 (각 1/3 확률)

```java
public static DispatchStatus random() {
    DispatchStatus[] statuses = values(); // [SUCCESS, FAILURE, ERROR]
    int index = ThreadLocalRandom.current().nextInt(statuses.length);
    return statuses[index];
}
```

`ERROR` 선택 시 `DispatchException` → HTTP 500 반환. 즉, **설계된 HTTP 에러율 = 33.3%**.

> `FAILURE`는 외부 서비스가 처리는 했지만 실패한 케이스(응답 body에 FAILURE 상태, HTTP 200)로, k6의 `http_req_failed`에는 집계되지 않는다.

### Tomcat 설정

| 항목 | 값 |
|------|----|
| max-threads | 200 (기본값) |
| accept-count | 100 |
| max-connections | 8192 |

## 3. 테스트 전략

**Breakpoint Test**: VU를 단계적으로 올려가며 서버가 버티는 한계 지점을 탐색한다.

```
50 VU  → (30s ramp-up + 1m hold)
100 VU → (30s ramp-up + 1m hold)
200 VU → (30s ramp-up + 1m hold)
300 VU → (30s ramp-up + 1m hold)
400 VU → (30s ramp-up + 1m hold)
       → 30s cooldown
```

총 소요 시간: 약 8분

> **VU(Virtual User)**: 쉬지 않고 연속적으로 요청을 보내는 가상 클라이언트.
> VU=200이면 200개의 요청이 항상 처리 중인 상태를 의미한다.

**임계치 판단 기준 (thresholds)**:

```javascript
http_req_duration: [
    'p(95)<15000',   // P95 응답시간 15초 이내 (자연 분포 P95≈12s 대비 여유)
    'p(99)<25000',   // P99 응답시간 25초 이내 (자연 분포 P99≈18s 대비 여유)
],
http_req_failed: [
    'rate<0.05',     // 에러율 5% 미만 (과부하 시 추가 에러 감지 목적)
],
```

에러율 threshold를 5%로 설정한 이유는 설계된 33% 에러가 아닌, **과부하로 인한 추가 에러(커넥션 거절, 타임아웃)가 발생하는지**를 감지하기 위함이다.

## 5. 분석

### 5-1. VU 200에서 TPS 상한 도달 — 이론값과 일치

<!-- 이미지 삽입: TPS 포화 구간 확대 그래프 -->

VU 200을 기점으로 TPS가 ~50/s에서 더 이상 증가하지 않는다. VU를 300, 400으로 올려도 동일하다.

```
실측: Tomcat 200 threads ÷ avg 응답시간 4s = 50 TPS ← 이론값과 정확히 일치
```

VU 200은 Tomcat 쓰레드 풀(200개)이 전부 점유되는 지점이다. 이를 초과하는 요청은 Tomcat NIO Connector가 TCP 연결 자체는 수락한 뒤(max-connections=8192 범위 내) 내부적으로 쓰레드를 기다리며 대기한다. 먼저 처리가 끝난 쓰레드가 즉시 다음 요청을 이어받는 구조다.

> `accept-count`는 max-connections(8192)마저 초과했을 때 OS TCP 레벨에서 사용하는 backlog 큐다. 이번 테스트는 최대 400 VU로 8192와는 거리가 멀어 실질적으로 관계없다.

### 5-2. VU 200 초과 시점부터 avg 응답시간이 증가하기 시작

200 VU까지는 avg 응답시간이 이론값인 4s 수준으로 유지됐다. 그러나 200 VU를 넘는 순간부터 avg가 5s대로 올라가기 시작했고, 이것이 임계치 도달의 핵심 신호다.

**이유**: 200 VU를 초과하는 순간부터 쓰레드 풀이 꽉 찬 상태에서 요청이 Executor 내부 큐에서 대기하게 된다. 이 큐 대기 시간이 처리 시간에 더해지면서 avg 응답시간이 올라가는 것이다.

```
200 VU 이하: avg ≈ 4s  (순수 처리 시간, 큐 대기 없음)
200 VU 초과: avg ≈ 5s+ (처리 시간 4s + 큐 대기 시간)
```

응답시간 증가 폭 자체는 크지 않아 보일 수 있지만, 이는 큐 회전이 빠르기 때문이다. 쓰레드 200개가 평균 4s짜리 요청을 처리하면 초당 50개씩 완료된다. 대기 요청이 50개 쌓여 있어도 평균 대기는 약 1s 수준이다.

그러나 이 1s 증가가 의미하는 바는 명확하다. **VU 200이 이 서버의 처리 한계이며, 이를 초과하면 응답시간은 VU에 비례해 계속 늘어난다.** 요청이 더 몰릴수록 큐가 깊어지고 대기 시간이 길어지는 구조다.

### 5-3. 에러율 분석 — 서버 측 ≈ k6 측, 과부하 거절 없음

<!-- 이미지 삽입: 서버 측 에러율 vs k6 측 에러율 비교 그래프 -->

테스트 전 구간에 걸쳐 서버 측 에러율과 k6 측 에러율이 약 33%로 거의 동일했다.

이 두 값이 일치한다는 것은 중요한 의미를 가진다.

만약 Tomcat 수락 큐가 넘쳐 **커넥션 거절(Connection Refused)** 이나 **타임아웃**이 발생했다면, 서버가 처리조차 하지 못한 요청이 k6에는 에러로 집계되지만 서버 메트릭(`http_server_requests_seconds`)에는 잡히지 않는다. 이 경우 **k6 에러율 > 서버 에러율** 이 된다.

```
서버 측 에러율 ≈ k6 측 에러율 ≈ 33%
→ k6가 본 모든 에러는 서버가 처리하고 반환한 HTTP 500
→ 과부하로 인한 커넥션 거절 0건
```

Tomcat accept-count(100) + max-connections(8192)가 400 VU의 요청은 여유롭게 수용했다는 의미다. 응답이 느려지더라도 요청 자체는 다 받아서 처리했다.

에러율 33.38%는 오롯이 `DispatchStatus.random()`이 `ERROR`를 선택한 경우로, **서버 설계상 의도된 값**이다.

### 5-4. cooldown 구간의 avg 응답시간 폭등

VU가 0으로 줄어드는 cooldown 구간에서 avg 응답시간이 최대 28s까지 치솟았다. 이를 서버 과부하로 오해할 수 있으나 원인은 다르다.

```
테스트 진행 중:
  짧은 요청(1~5s)들이 계속 완료되며 평균을 4s대로 유지

cooldown 진입 후:
  새 요청 유입 중단 → 짧은 요청은 이미 다 끝난 상태
  남아있는 건 지수 분포 상위 꼬리, 즉 20~30s짜리 롱테일 요청만
  → 이 요청들이 완료되면서 avg를 끌어올림
```

busy_threads가 감소하면서 동시에 avg 응답시간이 올라가는 현상은 얼핏 모순처럼 보이지만, **"처리 중인 쓰레드가 줄었다"는 것과 "남은 요청이 오래 걸린다"는 것은 독립적인 사실**이다.

## 6. 결론

### Mock 서버 기준 적정 동시 처리 한계: 200 VU

| 지표 | 임계 지점 | 근거 |
|------|----------|------|
| TPS 상한 | **200 VU (50 req/s)** | 이후 VU 증가해도 TPS 증가 없음 |
| 쓰레드 포화 | **200 VU (200 threads)** | Prometheus `tomcat_threads_busy_threads` 실측 |
| 응답시간 | 변화 없음 (4s 유지) | VU 50~400 전 구간 동일, 큐 대기 0.02s 수준 |
| 과부하 거절 | **없음** | k6 에러율 ≈ 서버 에러율 = 33% (설계값) |

### 미들웨어 설계에 주는 시사점

이번 테스트가 드러낸 본질적인 문제는 쓰레드 풀 크기 자체가 아니다.

외부 알림 API를 **동기 방식으로 호출하는 구조**에서는 Tomcat 쓰레드가 외부 API 응답을 기다리는 동안 점유 상태로 묶인다. 이 서버처럼 평균 응답시간이 4초라면, 쓰레드 1개당 초당 0.25건 밖에 처리하지 못한다. 쓰레드 200개로도 50 TPS가 한계다.

외부 API 응답시간이 늘어날수록, 동시 사용자가 많아질수록 이 한계는 더 빠르게 드러난다.

```
처리량을 늘리는 방법:
  1. 비동기 처리 — 응답 반환 후 별도 쓰레드에서 발송 (쓰레드 점유 시간 단축)
  2. 메시지 큐   — 발송 요청을 큐에 넣고 Consumer 워커가 처리 (백프레셔 자연스럽게 처리)
  3. max-threads 증가 — 임시방편. 외부 API가 느려질수록 필요 쓰레드 수가 선형 증가

근본 해결책은 1번 또는 2번.
```

이 Mock 서버 부하테스트는 "동기 방식의 구조적 한계를 수치로 확인하고, 비동기 또는 큐 기반 아키텍처로 전환하는 근거를 마련하기 위한 사전 검증"이었다.

## 부록: k6 오류율 메트릭 주의사항

k6를 Prometheus remote write로 연동할 때 `k6_http_req_failed_rate` 메트릭은 `expected_response` 라벨로 두 개의 시리즈로 분리된다.

| 시리즈 | 값 |
|--------|-----|
| `k6_http_req_failed_rate{expected_response="true"}` | ≈ 0 (2xx 요청은 실패하지 않음) |
| `k6_http_req_failed_rate{expected_response="false"}` | ≈ 1 (5xx 요청은 항상 실패) |

이 상태에서 `avg(k6_http_req_failed_rate) * 100` 또는 `sum/count` 방식으로 집계하면 항상 **50%**가 나온다. 실제 에러율과 무관한 잘못된 값이다.

올바른 k6 전체 오류율 쿼리:

```promql
sum(rate(k6_http_reqs_total{expected_response="false"}[30s]))
  / sum(rate(k6_http_reqs_total[30s])) * 100
```

카운터(`k6_http_reqs_total`) 기반으로 실패 요청 비율을 직접 계산하는 방식이 정확하다.
